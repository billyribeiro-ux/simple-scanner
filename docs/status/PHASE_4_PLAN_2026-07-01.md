# Phase 4 Plan - Target Runtime Bring-Up, Migration Verification, And Persisted API Smoke Testing

Date: 2026-07-01

## Objective

Bring the target backend runtime as far online as the local machine allows, verify PostgreSQL/TimescaleDB migrations when the Python `3.14.6` venv is available, and prove the persisted API vertical slice with a deterministic mocked workflow that does not require FMP, internet, or secrets.

## Scope Guardrails

- No UI expansion or redesign.
- No broker execution, order routing, options, gamma, Greeks, IV, market internals, or WebSocket work.
- No new ML work and no self-learning claim.
- No profitability claim.
- No secret logging, export, persistence, or documentation.

## Implementation And Verification Sequence

1. Runtime discovery
   - Verify `.python-version` is `3.14.6` and `.node-version` is `24.18.0`.
   - Inspect `python3.14`, `pyenv`, `asdf`, `uv`, `brew`, and OS package availability.
   - Attempt a safe user-space Python `3.14.6` install only if a safe manager is already available.
   - If Python `3.14.6` remains unavailable, document exact install commands and keep setup failure clean/actionable.

2. Runtime command hardening
   - Update `docs/runtime-setup.md` with OS-specific Python `3.14.6` install options.
   - Extend `scripts/doctor.sh` to report active repository backend selection honestly.
   - Add optional `make fmp-smoke` only as a gated, redacted manual command.

3. Persistence backend contract
   - Inspect and document the repository registry behavior.
   - Make `/health` and `/config` report safe persistence backend fields.
   - Be explicit that API runtime is SQLite-backed unless `DATABASE_URL` is `sqlite:///...`; PostgreSQL is migration-verified but not yet repository-backed if no Postgres repository implementation exists.

4. Backend venv and dependency verification
   - If `python3.14` exists, run `make setup-backend`.
   - Verify backend venv tool versions: Python, pip list, pytest, ruff, mypy, alembic, uvicorn.
   - Run `make backend-test`, `make backend-lint`, and `make backend-typecheck` through the venv.
   - If blocked, record exact failure output and do not use Python 3.9 as target runtime.

5. Docker and migrations
   - Run `docker compose config`, `docker compose up -d postgres redis`, and `docker compose ps`.
   - Run `make db-migrate` only when the backend venv exists.
   - Add a schema inspection script/test that can verify expected tables, indexes, and constraints after migration when the target runtime is available.

6. Persisted API smoke workflow
   - Add a deterministic smoke script and/or pytest using synthetic bars and mocked providers.
   - The smoke must avoid FMP and internet.
   - It must persist bars, features, candidate signals, labels, validation reports, model runs, active models, scanner runs/signals, exports, and daily reviews.
   - It must prove rejected activation fails, accepted activation succeeds, and persisted data survives repository reinitialization.

7. Export and security verification
   - Verify CSV and XLSX exports read persisted signals and record export metadata.
   - Verify generated files live under ignored `exports/`.
   - Run a secret scan over repo/runtime outputs and document the result.

8. Frontend non-regression
   - Run `corepack pnpm check`, `corepack pnpm build`, `corepack pnpm test`, and `corepack pnpm lint`.
   - Do not edit frontend unless backend contract changes break TypeScript.

9. Documentation and handoff
   - Create `docs/api-smoke-testing.md`.
   - Create `docs/status/PHASE_4_RUNTIME_VERIFICATION_2026-07-01.md`.
   - Create `docs/status/PHASE_4_COMPLETION_2026-07-01.md`.
   - Update `README.md`, `docs/HANDOFF.md`, `docs/runtime-setup.md`, `docs/persistence-architecture.md`, `docs/data-model.md`, and methodology docs only where behavior changed.

## Completion Criteria

Phase 4 is complete when all practical runtime commands are attempted, target-runtime blockers are documented exactly, Docker services are verified, migrations run or are honestly blocked, the persisted smoke workflow passes without secrets, exports are reproducible from persisted state, frontend gates still pass, and final docs state the active API persistence backend without ambiguity.
