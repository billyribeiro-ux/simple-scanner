# Phase 18 Plan - Runtime FMP Key Bring-Up And Live Seed Verification

Date: 2026-07-03
Base commit: `65a488d3aaed13b28b690e8210f46eb467252482`

## Scope

Phase 18 is an operator live-verification phase, not a feature-expansion phase. The goal is to bring up `FMP_API_KEY` from runtime-only secret handling, rerun live entitlement, review measured endpoint rows, and run bounded real-data seed/freshness checks only when entitlement review allows.

No broker execution, order routing, options analytics, market internals, Level 2, dark-pool or order-book data, production WebSocket ingestion, automatic activation, self-learning claims, or profitability claims are in scope.

## Secret Handling Plan

- Keep the key out of committed files, docs, exports, logs, frontend bundles, provider metadata, scheduler payloads, and model artifacts.
- Use runtime environment injection only for live commands. Do not echo the value.
- Keep `.env` and `.env.local` ignored; do not stage either file.
- Verify `make doctor` reports key presence only.
- Verify the FMP REST client still uses header auth and strips query-string `apikey` params.
- Run exact-value and generic secret scans after live/export/frontend work.

## Live Verification Sequence

1. Runtime preflight:
   - verify branch starts from `origin/main`;
   - verify env files are ignored;
   - run Node/pnpm/Python/doctor/database setup gates.
2. Live entitlement:
   - run `make fmp-smoke`;
   - run `make fmp-live-smoke`;
   - run `POST /provider/capabilities/check` for `SPY,QQQ,AAPL,NVDA` and required endpoint keys.
3. Operator review:
   - mark only usable non-empty `ACCESSIBLE` rows as `REVIEWED_ACCESSIBLE`;
   - mark denied/rate-limited/unusable rows honestly;
   - read review summary and proceed only if `READY`.
4. Seed workflow:
   - run seed dry-run first;
   - run live seed only if review summary is `READY`;
   - record quote snapshot, EOD, intraday, provider request, ingestion run, dirty-window, warning, and error counts.
5. Incremental refresh:
   - after successful live seed, run bounded incremental intraday refresh twice for `SPY,QQQ,AAPL,NVDA` at `1min,5min,15min`;
   - verify no duplicate bar growth on the second run.
6. Freshness and research gates:
   - persist freshness check and read latest report;
   - create controlled research-cycle dry-run using real ingested symbols when live data exists;
   - verify default block and `allow_stale=true` warning behavior where practical.
7. Scheduler jobs:
   - run `fmp_capability_check`;
   - run `fmp_seed_ingestion` dry-run;
   - run live `fmp_seed_ingestion` only if review summary remains `READY` and rate-limit risk is acceptable;
   - run `data_freshness_check`;
   - run `fmp_incremental_intraday_refresh` if live seed succeeded.
8. Exports/UI:
   - export entitlement review, capability matrix, quote snapshots, seed ingestion, freshness, and data coverage;
   - run Playwright e2e and record operator UI status.
9. Regression:
   - run backend, database, frontend, scheduler, export, compile, diff, and secret-scan gates requested by the chief architect.

## Stop Conditions

- If key presence cannot be established without exposing the key, stop live execution and document `BLOCKED-NO-RUNTIME-KEY`.
- If required endpoints are denied, rate-limited, empty, or unusable, do not run live seed except for an explicit blocked negative test.
- If live responses reveal parser, timestamp, idempotency, entitlement-classification, redaction, freshness, scheduler, export, or UI defects, make narrow fixes only and rerun targeted tests.
