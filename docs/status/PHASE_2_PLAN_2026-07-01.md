# Phase 2 Plan — Quant Core Hardening and No-Leakage Foundation

Report date: 2026-07-01

## Objective

Make the quant core falsifiable, testable, and no-leakage by prioritizing deterministic backend modules over UI work. This phase will not claim the platform is self-learning or production-ready. It will create the foundation needed for serious model work.

## Implementation Sequence

1. Runtime and command hardening
   - Keep `.node-version` at `24.18.0`.
   - Keep `.python-version` at `3.14.6`.
   - Add `make doctor` with checks for Node, Corepack, pnpm, Python 3.14, backend venv, Docker daemon, `FMP_API_KEY` presence, and database URL presence without printing secrets.
   - Add `make setup-backend` for Python 3.14.6 venv creation.
   - Add `make quant-test` for pure quant tests that do not require FMP, Docker, Redis, or Postgres.
   - Avoid the old Python 3.9 `datetime.UTC` collection failure in the pure quant path where practical, while still documenting that the supported target is Python 3.14.6.

2. Pure quant schemas
   - Add `services/quant-engine/app/quant/types.py`.
   - Define database-free classes for `Bar`, `FeatureRow`, `LabelRow`, `CandidateSignal`, `SimulatedTrade`, `ValidationSplit`, `WalkForwardWindow`, `ValidationReport`, `ModelRun`, and `SignalDecision`.
   - Keep these objects importable in pure tests without FastAPI, FMP, Docker, or the database.

3. Grouped no-leakage feature engine
   - Refactor `services/quant-engine/app/features/engine.py`.
   - Group calculations by `symbol`, `interval`, and `session_date`.
   - Add data quality flags for duplicates, missing OHLCV, invalid prices, negative volume, and timezone issues.
   - Implement grouped VWAP, true range/ATR, rolling relative volume, same-time-of-day relative volume from prior sessions, previous-day levels, premarket levels, opening range completion logic, gap features, candle/range/wick metrics, trend/slope features, volume acceleration, absorption proxy, liquidity sweep proxy, failed breakout/breakdown flags, relative strength vs SPY/QQQ, and time buckets.
   - Preserve compatibility for existing API callers by returning dictionaries from `build_features`.

4. Candidate signal layer
   - Add `services/quant-engine/app/signals/candidates.py`.
   - Generate deterministic `CandidateSignal` rows from current/past feature rows only.
   - Cover VWAP reclaim/loss, opening range breakout/breakdown, premarket breaks, previous-day levels, liquidity sweep reversals, failed breakouts/breakdowns, and trend continuations.
   - Generate inspectable reason and warning codes.

5. No-leakage label engine
   - Refactor `services/quant-engine/app/labels/engine.py`.
   - Add configurable label settings for max hold, entry mode, target R, stop mode, ATR multiplier, slippage/spread, same-bar policy, overlap policy, and minimum warmup bars.
   - Use next-bar-open entries by default.
   - Compute stops and targets only from candidate-time information.
   - Use future bars only for outcome measurement.
   - Return existing Pydantic labels for API compatibility while using internal `LabelRow` logic.

6. Chronological backtest simulator
   - Replace label-only summary behavior in `services/quant-engine/app/backtesting/engine.py` with a deterministic trade-metric engine over labeled/candidate outcomes.
   - Add support for chronological ordering, one open trade per symbol, conservative same-bar handling, time exits, drawdown, profit factor, MFE/MAE, and breakdowns.
   - Keep `summarize_labels` available for route compatibility.

7. Validation engine and activation gate
   - Add `services/quant-engine/app/validation/engine.py`.
   - Implement chronological splits, walk-forward windows, embargo handling, leakage checks, validation reports, and activation acceptance/rejection.
   - Use this gate from the model layer instead of weak average-R-only activation.

8. Baseline evidence model
   - Refactor `services/quant-engine/app/models/engine.py`.
   - Compute evidence by symbol, setup, regime, time bucket, and side.
   - Include sample size, win rate, average/median R, profit factor, drawdown, hit rates, MFE/MAE, validation metadata, and activation decision in artifacts.
   - Keep ML optional and do not add a black-box model in this phase.

9. Scanner historical context
   - Refactor `services/quant-engine/app/jobs/scanner.py`.
   - Hydrate recent intraday history before scoring.
   - Maintain per-symbol context buffers.
   - Generate NO_TRADE with warnings when context is insufficient.
   - Keep tests provider-mocked and avoid live FMP requirements.

10. Persistence path without blocking quant work
    - Reconcile obvious schema/migration divergence for feature/label/signal/model fields where practical.
    - Add repository interfaces for bars, features, labels, signals, and model runs.
    - Keep pure quant tests independent of Docker/Postgres.

11. Tests
    - Add synthetic fixtures for trend, chop, opening range breakout, liquidity sweep, failed breakout, VWAP reclaim, missing bars, duplicates, mixed symbols, and mixed sessions.
    - Add pure quant tests for grouped features, labels, backtest metrics, validation windows/gates, candidate detection, scanner context, and secret redaction.

12. Documentation and verification
    - Update `README.md`, `docs/HANDOFF.md`, `docs/validation-methodology.md`, `docs/signal-methodology.md`, and create `docs/quant-core-performance.md`.
    - Create `docs/status/PHASE_2_COMPLETION_2026-07-01.md`.
    - Run practical commands: `make doctor`, `make quant-test`, compileall, backend pytest if possible, backend ruff/mypy if available, `docker compose config`, `corepack pnpm check`, and `corepack pnpm build`.

## Scope Boundaries

- No frontend redesign or UI polish.
- No options, gamma, Greeks, IV, market internals, or broker execution.
- No profitability claims.
- No self-learning claim until daily review, controlled retraining, validation, and activation rules are complete.

## Expected End State

Phase 2 is complete only if pure quant tests run without FMP/Docker and the code can produce a chronological validation report with explicit activation acceptance or rejection. If local Python 3.14.6 remains unavailable, full backend runtime verification will remain blocked and documented.
