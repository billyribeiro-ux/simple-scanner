# Phase 3 Plan - Backend Runtime and Persistence Hardening

Date: 2026-07-01

## Objective

Replace API-level in-memory workflow state with repository-backed persistence, harden local backend runtime commands, and document the actual run state without expanding V1 scope beyond scanner, research, backtest, validation, model activation gates, signals, and exports.

## Scope Guardrails

- No broker execution, order routing, options, gamma, Greeks, internals, or WebSocket expansion.
- No new ML family, self-learning claim, profitability claim, or production-readiness claim.
- Secrets remain environment-only. The FMP key is never written, printed, logged, exported, bundled, or committed.
- Temporary in-memory objects may be used only inside tests or event streaming plumbing, not as the API source of truth.

## Work Plan

1. Runtime command hardening
   - Add make targets for `help`, `backend-test`, `backend-lint`, `backend-typecheck`, `db-down`, `db-reset-dev`, `api-dev`, `web-dev`, and `dev`.
   - Expand `scripts/doctor.sh` to check Node `24.18.0`, Corepack/pnpm, Python `3.14.6`, backend venv, installed backend tools, Docker/database reachability, ignored env/data paths, `DATABASE_URL`, and `FMP_API_KEY`.

2. Persistence foundation
   - Replace placeholder repository protocols with concrete repository classes backed by a durable local SQLite store for local-first API runtime and tests.
   - Keep PostgreSQL/TimescaleDB as the documented production target and align SQLAlchemy metadata plus Alembic migration with the Phase 3 table set.
   - Cover symbols, bars, features, candidate signals, labels, validation reports/windows, model runs/artifacts, active models, live/closed signals, scanner runs, provider requests, exports, and daily reviews.

3. Workflow services
   - Add service classes for ingestion, feature building, labeling, validation, model training, activation, backtesting, scanner persistence, exports, and daily review payloads.
   - Keep quant logic delegated to existing engines.
   - Add activation guards that require accepted validation evidence before setting an active model.

4. API route conversion
   - Remove module-level `_MEMORY` as route source of truth.
   - Convert data, feature, label, model, backtest, scanner signal, export, and review routes to repositories/services.
   - Keep SSE queue as transient delivery plumbing only.

5. Tests
   - Add persistence tests using temporary SQLite files.
   - Add service/API-adjacent workflow tests for repository-backed ingestion, feature/label persistence, activation guard behavior, scanner signal persistence, and export metadata.
   - Preserve existing pure quant tests.

6. Documentation and handoff
   - Add `docs/persistence-architecture.md`.
   - Add `docs/runtime-setup.md`.
   - Update `docs/HANDOFF.md`.
   - Add `docs/status/PHASE_3_COMPLETION_2026-07-01.md` with commands, results, blockers, and remaining quant risks.

Phase 3 is complete when the API source of truth is repository-backed, runtime setup clearly reports missing local prerequisites, migrations/schema match the repository contract, tests cover the new persistence layer, documentation is updated, and any local runtime blockers are explicitly documented.
