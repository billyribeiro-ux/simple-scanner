# Phase 6 Plan - Candidate-To-Trade Market Replay And Incremental Pipeline Hardening

Date: 2026-07-01

## Objective

Add a real candidate-to-trade market replay mode that replays persisted candidate signals through raw bars chronologically, persists replay runs and simulated trades, exposes replay APIs and exports, and preserves the existing label-derived evidence mode as a separate simulation type.

## Scope Guardrails

- No frontend UI or dashboard redesign.
- No broker execution, order routing, WebSocket scope, options, gamma, Greeks, IV, or market internals.
- No calibrated ML, self-learning language, or profitability claims.
- No secrets in docs, routes, logs, exports, or persisted provider metadata.
- Preserve the Phase 5 SQLite/Postgres repository architecture and explicit backend selection.

## Implementation Sequence

1. Add replay domain types and engine
   - Create `app/backtesting/replay.py`.
   - Define `ReplayConfig`, `ReplayRun`, `ReplayDecision`, `SimulatedTrade`, `ReplayMetrics`, `ReplaySkipReason`, `IntrabarPolicy`, and `ExecutionAssumption`.
   - Replay persisted candidates against raw bars by symbol/interval/time, using next-bar-open entry by default.
   - Implement stop/target/time/session exits, conservative same-bar ambiguity handling, slippage/spread adjustments, overlap limits, portfolio limits, cooldowns, deterministic candidate priority, MFE/MAE, and metrics from simulated trades only.

2. Add replay persistence schema
   - Extend SQLAlchemy metadata with `replay_runs`, `simulated_trades`, and a small `pipeline_build_windows` metadata table for incremental/stale tracking.
   - Add Alembic revision `0003_phase6_replay`.
   - Extend SQLite initialization with equivalent tables and indexes.
   - Update expected table/revision/index/JSON-column checks for `make db-inspect`.

3. Add repository methods
   - Add repositories for replay runs, simulated trades, and pipeline build windows.
   - Support bulk trade persistence, skipped-candidate persistence through `simulated_trades`, paginated trade queries, replay summaries, and SQLite/Postgres parity.
   - Keep safe JSON serialization and no secret fields.

4. Add services and routes
   - Keep `POST /backtest/run` as label-derived and explicitly return `simulation_type = label_derived`.
   - Add `POST /backtest/replay`, `GET /backtest/replay/{replay_run_id}`, and `GET /backtest/replay/{replay_run_id}/trades`.
   - Make `GET /backtest/runs` and `GET /backtest/runs/{run_id}` combine label-derived and replay runs with explicit `simulation_type`.

5. Add replay exports
   - Add persisted replay summary XLSX, simulated trades CSV, simulated trades XLSX, and replay metrics JSON.
   - Workbook sheets: Summary, Trades, Skipped Candidates, Per Symbol, Per Setup, Per Regime, Per Time Bucket, Daily R, Drawdown, Config, and Warnings.
   - Record export metadata and source replay run IDs.

6. Add incremental pipeline hardening
   - Record dirty bar windows on bar upsert by symbol/interval/session/date range.
   - Add build-window metadata for features, candidates, labels, and replay awareness.
   - Return stale/dirty status from feature and label build services.
   - Limit feature/candidate/label rebuilds to requested symbols/intervals/time ranges plus local warmup where practical.

7. Add replay-aware validation integration
   - Add `validation_mode` support with default `label_derived`.
   - Allow replay validation reports to be saved and used for activation only when explicitly requested.
   - Include `validation_mode` in activation responses and docs.

8. Add tests
   - Pure replay unit tests for long/short wins/losses, time exit, same-bar ambiguity, missing entry, invalid risk, overlap, portfolio limit, cooldown, priority, slippage/spread, MFE/MAE, drawdown, profit factor, daily series, and breakdowns.
   - Repository parity tests for replay runs/trades/build windows on SQLite and Postgres.
   - API tests for replay routes and combined backtest run listing.
   - Export tests for replay workbook/CSV/JSON output and metadata.
   - Incremental tests proving unrelated symbols are not rebuilt for narrow ranges.

9. Update docs
   - Update README, HANDOFF, data model, signal methodology, validation methodology, quant performance, API smoke, and persistence docs.
   - Create `docs/market-replay-methodology.md`, `docs/status/PHASE_6_COMPLETION_2026-07-01.md`, and `docs/status/PHASE_6_REPLAY_VALIDATION_2026-07-01.md`.

10. Verify and commit
   - Run the required backend, database, parity, replay, export, frontend, compile, diff, and secret-scan gates.
   - Commit only source/docs/tests/migrations, never env files, runtime DBs, exports, model artifacts, venvs, node modules, or secrets.

## Completion Criteria

Phase 6 is complete when replay metrics are computed from replayed candidate trades rather than labels, replay data persists across SQLite and Postgres, replay APIs and exports read persisted replay data, label-derived mode remains explicit and separate, incremental metadata reports dirty/stale windows, docs state remaining quant limits honestly, and all practical verification commands pass.
