# Phase 12 Plan

Status date: 2026-07-01

Phase 12 brings the frontend onto the exact target runtime, then adds a thin operator governance UI for the Phase 11 research cycle and proposal lifecycle. It must remain a control surface for human review only: no broker execution, no order routing, no automatic activation, no self-learning language, no secret exposure, and no profitability claims.

## Implementation Sequence

1. Verify and repair the target frontend runtime:
   - confirm `.node-version` and root `package.json` target Node `24.18.0`;
   - inspect `node`, `corepack`, `pnpm`, `nvm`, `fnm`, `asdf`, `volta`, `mise`, and `brew`;
   - use local NVM by sourcing `~/.nvm/nvm.sh`;
   - install/use Node `24.18.0` through NVM if it is not already installed;
   - run `COREPACK_ENABLE_DOWNLOAD_PROMPT=0 corepack prepare pnpm@11.9.0 --activate`;
   - verify `node --version` and `corepack pnpm --version`.

2. Add `make frontend-doctor`:
   - verify Node exactly `24.18.0`;
   - verify Corepack and pnpm;
   - verify workspace packages and SvelteKit dependency;
   - verify frontend strict TypeScript config;
   - scan frontend source for disallowed secret references.

3. Extend shared/frontend types and API client:
   - add shared Zod schemas/types for research cycles, artifacts, model proposals, decision-ledger events, research status, export metadata, activation responses, and API errors;
   - make `PUBLIC_API_BASE_URL` the safe frontend API base override with local default;
   - add typed functions for health/config/status, cycles, proposals, ledger, and Phase 11 exports;
   - preserve non-secret error handling.

4. Build thin operator UI routes:
   - `/operations` for health/config/research status;
   - `/research/cycles` for cycle list/create/dry-run/run/export;
   - `/research/cycles/[id]` for cycle detail/artifacts/export;
   - `/research/proposals` for proposal list;
   - `/research/proposals/[id]` for detail, approve, reject, and explicit activation;
   - `/research/decision-ledger` for filterable ledger review;
   - `/research/status` as a read-only research status page.

5. Enforce UI guardrails:
   - approve button must only call approve;
   - activation must require a visible confirmation panel and a typed phrase before sending `confirm_manual_activation=true`;
   - show backend blocked responses directly;
   - use copy such as “No broker execution” and “Scanner model update only”;
   - avoid buy/sell/order controls and any secret fields.

6. Add frontend E2E coverage:
   - operations page loads with mocked backend responses;
   - research cycles page loads and create form uses safe defaults;
   - dry-run button calls the expected endpoint;
   - proposal list/detail load;
   - approve does not activate;
   - activation cannot fire before explicit confirmation and does send confirmation afterward;
   - decision ledger filters work;
   - pages do not contain `FMP_API_KEY`, `DATABASE_URL`, or forbidden execution controls.

7. Update documentation:
   - README, HANDOFF, runtime setup, controlled research cycle, model proposal lifecycle, champion/challenger governance, decision ledger, API smoke testing, operational hardening;
   - create Phase 12 completion/runtime status docs, operator UI guide, and manual activation safety guide.
