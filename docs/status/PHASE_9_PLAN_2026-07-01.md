# Phase 9 Plan - Counterfactual Replay And Calibration Audit

Status date: 2026-07-01

## Scope

Phase 9 separates candidate-quality evidence from portfolio-style replay. It adds `model_training_counterfactual` replay, upgrades replay-aware training to prefer independent candidate outcomes, adds explicit replay-aware validation windows, persists score calibration audits, gates activation/scanner output on calibration when requested, and adds diagnostic comparisons. It does not add broker execution, order routing, WebSocket, options, market internals, black-box ML, self-learning claims, or profitability claims.

## Implementation Sequence

1. Add replay purpose configuration to `ReplayConfig` and `ReplayBacktestRequest`: `portfolio_execution` and `model_training_counterfactual`, explicit portfolio/symbol overlap enforcement flags, invalid-candidate capture, and the `candidate_quality_evidence` result label.
2. Extend `CandidateMarketReplayEngine` so counterfactual mode replays every valid candidate independently through the existing next-bar-open OHLCV simulation while skipping invalid/session/stale/data-quality failures. Persist purpose, constraint flags, concurrency, overlap group, and candidate-quality metadata in replay/trade JSON.
3. Upgrade replay metrics to label counterfactual runs as non-portfolio candidate-quality evidence and report valid/invalid/observed counts, concurrency density, overlap density, long/short/symbol/setup/regime/time breakdowns, and explicit warnings against treating counterfactual output as portfolio P/L.
4. Add Alembic revision `0006_phase9_counterfactual_calibration`, SQLAlchemy schema tables, SQLite initialization, repository constants, and db-inspect expectations for `model_calibration_audits`, `model_calibration_bins`, and `model_comparisons`.
5. Add repositories for calibration audits/bins and model comparisons with SQLite/Postgres parity and pagination/list/get helpers.
6. Update `CandidateOutcomeDatasetBuilder` to support `counterfactual_preferred`, `counterfactual_only`, `portfolio_only`, and `mixed_allowed`, preserving portfolio outcomes separately from counterfactual candidate-quality outcomes and excluding invalid/stale rows from observed evidence.
7. Update `replay_aware_baseline` training request/service metadata for counterfactual/portfolio replay IDs, outcome-source policy, counterfactual requirements, portfolio-only fraction warnings, overlap/concurrency filters, and artifact provenance.
8. Strengthen replay-aware validation to accept explicit training/validation/test replay IDs and windows, enforce embargo and counterfactual/portfolio requirements, score validation/test rows from training-only evidence, and persist window provenance.
9. Create `app/models/calibration_audit.py` to aggregate score audits against replay outcomes by score, grade, action, symbol, setup, regime, and time bucket, producing monotonicity/rank-correlation/separation metrics plus warnings/rejection reasons.
10. Add calibration audit API routes: create, list, get, and bins list. Add XLSX/CSV/JSON exports with Summary, Score Bins, Grade Bins, Action Bins, stability, warning, config, and provenance sheets.
11. Add activation gates for required calibration audits, monotonic score bins, TAKE vs WATCH separation, high-grade sample minimums, rank correlation, and maximum warning counts. Include calibration metadata in activation responses.
12. Add scanner guardrails so calibration-required active replay-aware models suppress actionable TAKE output when calibration is missing or failed, while emitting non-secret status/warnings/reasons in the signal payload.
13. Add diagnostic model comparison (`POST /models/compare`) and counterfactual-vs-portfolio comparison (`POST /backtest/compare-counterfactual-vs-portfolio`, `GET /backtest/counterfactual-comparisons/{id}`), persisted as research artifacts only.
14. Update exports and API smoke coverage for counterfactual replay, calibration audits, model comparison, and counterfactual-vs-portfolio comparison.
15. Add focused regression tests for replay mode behavior, dataset source priority, training/validation requirements, calibration pass/fail warnings, activation/scanner gates, persistence parity, exports, and API routes.
16. Update README, HANDOFF, methodology docs, data model, persistence, API smoke, performance, and Phase 9 status docs.
