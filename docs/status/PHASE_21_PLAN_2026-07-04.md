# Phase 21 Plan - 2026-07-04

`PHASE_21_STATUS = PLANNED_DIAGNOSTIC_ONLY`

Phase 21 is a challenger rejection diagnostics, gate-attribution, and evidence-improvement phase. It is not a model-activation phase, not a feature-expansion phase, not a gate-loosening phase, and not a live-ingestion expansion phase.

## Guardrails

- Do not activate models or call `confirm_manual_activation`.
- Do not approve proposals automatically.
- Do not loosen validation, calibration, model-review, stale, or freshness gates.
- Do not add broker execution, order routing, options, market internals, Level 2, dark-pool data, production WebSocket ingestion, black-box ML, or self-learning behavior.
- Do not make profitability claims.
- Do not expose `FMP_API_KEY` or place keys in query strings.
- Prefer persisted Phase 19D/20 evidence; no live FMP call is required for this diagnostic phase.

## Evidence Collection

Collect exact persisted IDs and payloads for:

- Phase 20 research cycle.
- Challenger model version.
- Replay-aware validation report.
- Calibration audit.
- Model review report.
- Champion/challenger comparisons.
- Model proposals.
- Decision-ledger rows.
- Portfolio and counterfactual replay runs.
- Phase 20 exports.

Missing Phase 20 subreport documents will be recorded as documentation gaps, not inferred evidence.

## Diagnostic Work

1. Attribute each rejection/blocking reason to persisted artifacts.
2. Analyze data, replay, evidence/model, validation, calibration, drift/model-review, proposal, and decision-ledger gate families.
3. Extract validation and calibration bucket diagnostics without treating scores as probabilities.
4. Analyze sample size, selected-candidate scarcity, high-grade counts, symbol/setup/day/time/regime concentration, ambiguity, and sensitivity evidence.
5. Compare `model_training_counterfactual` evidence with `candidate_market_replay` evidence and identify portfolio-constraint drag.
6. Prepare an evidence-improvement plan that keeps the challenger rejected until stronger evidence exists.
7. Run required regression and secret-scan commands, using isolated databases for mutating tests where needed so the certified evidence database is not contaminated.
