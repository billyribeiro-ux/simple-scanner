# Phase 7 Plan - Replay Calibration Audit, Sensitivity Analysis, And Operational Hardening

Date: 2026-07-01

## Objective

Make candidate market replay auditable, reproducible, sensitivity-tested, stale-aware, and explicit enough for future model-selection work without adding broker execution, frontend UI, options data, WebSockets, calibrated ML, self-learning language, or profitability claims.

## Scope Guardrails

- Keep `label_derived` and `candidate_market_replay` separate.
- Do not weaken SQLite/Postgres repository parity.
- Do not expose `FMP_API_KEY`, database URLs, passwords, environment values, local secrets, or raw provider credentials in hashes, metadata, exports, docs, or routes.
- Keep REST/FMP entitlement work out of scope unless a live smoke can run safely from environment.
- No frontend redesign or new frontend UI.

## Implementation Sequence

1. Add Phase 7 persistence and inspection
   - Add Alembic revision `0004_phase7_audit`.
   - Add replay provenance columns where useful and new tables for sensitivity runs, sensitivity scenarios, and label-vs-replay comparisons.
   - Extend SQLite initialization with matching tables/indexes.
   - Update `EXPECTED_TABLES`, expected revision, db-inspect table/index/JSON-column checks, and Timescale status reporting.
   - Add safe Timescale hypertable attempt for `bars` only when supported, with no SQLite impact.

2. Add replay provenance and fingerprinting
   - Create deterministic hashing helpers for replay config, bars, features, candidates, and safe replay metadata.
   - Include replay config version, feature version, candidate version, label config version, source counts, backend, migration revision, git commit, config hash, input fingerprint, candidate fingerprint, and stale status in replay payloads.
   - Ensure no env values or secrets enter fingerprints.

3. Harden stale-window workflow
   - Add pipeline status query support and `GET /pipeline/status`.
   - Keep bar upserts marking feature/candidate/label/replay windows dirty.
   - Make replay reject dirty feature/candidate windows by default; allow only with `allow_stale = true` and persisted warnings.
   - Make replay validation reject stale replay runs by default unless `allow_stale_replay_validation = true`.

4. Add explicit replay validation selection
   - Extend validation service/API to accept `replay_run_id`, `replay_filter`, and `allow_latest_replay_fallback`.
   - Persist selected replay run ID, selection reason, config hash, input fingerprint, and replay metrics in validation reports.
   - Include replay run and sensitivity details in activation responses for replay validation.

5. Add sensitivity analysis
   - Create `app/backtesting/sensitivity.py`.
   - Rerun replay across compact grids for slippage, spread, same-bar policy, and intrabar policy.
   - Compute scenario metrics, worst/median/best cases, robustness score, deterioration summaries, fragility flags, and pass/fail gate decisions.
   - Persist sensitivity run summaries and compact scenario rows.

6. Add sensitivity APIs and exports
   - Add sensitivity create/get/list/scenario routes.
   - Add sensitivity summary XLSX, scenario CSV/XLSX, and metrics JSON exports.
   - Extend export metadata with file SHA-256, workbook sheets, source simulation type, config hash, input fingerprint, filters, and warnings where available.

7. Add label-vs-replay comparison
   - Add a persisted comparison report between label-derived backtest metrics and replay metrics.
   - Generate divergence flags for win rate, average R, profit factor, drawdown, trade count, symbol, and setup instability.
   - Add create/get routes and JSON export if practical.

8. Add operational diagnostics
   - Add `scripts/db_query_diagnostics.py` for safe, non-secret Postgres query diagnostics.
   - Update docs with Timescale hypertable/compression/retention notes and expected bottlenecks.

9. Add tests
   - Add provenance/fingerprint stability and no-secret tests.
   - Add explicit replay validation selection tests.
   - Add sensitivity engine, persistence, API, export, and gate tests.
   - Add label-vs-replay comparison tests.
   - Add stale-window gating and pipeline status tests.
   - Extend SQLite/Postgres repository parity and API smoke workflows.

10. Verify, document, commit, and push
   - Update README, HANDOFF, market replay methodology, validation methodology, signal methodology, data model, persistence architecture, API smoke, quant performance, and runtime setup.
   - Create Phase 7 completion, sensitivity, replay calibration audit, and operational hardening docs.
   - Run all practical backend, database, frontend, compile, diff, and secret-scan gates.
   - Commit and push only intended source, docs, tests, scripts, and migration files.

## Completion Criteria

Phase 7 is complete when replay runs are provenance-hashable, replay validation can select an explicit run, sensitivity runs persist and gate validation, label-derived and replay metrics can be compared, stale windows can block replay/validation by default, export manifests include hashes and source metadata, Postgres inspection reports operational structures, SQLite/Postgres parity still passes, docs state remaining limits honestly, and the practical command matrix passes.
