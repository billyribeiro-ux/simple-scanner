# Phase 19B Plan - Runtime Evidence Recovery Or Regeneration

Date: 2026-07-04

Phase 19B is a runtime evidence recovery/regeneration phase. It is not a feature-expansion phase and does not add broker execution, order routing, options workflows, production WebSocket ingestion, black-box ML, automatic model activation, self-learning behavior, or profitability claims.

## Objective

Move Phase 19 from `EVIDENCE_PENDING` only if real runtime evidence can be recovered or regenerated.

Allowed final outcomes:

- `RECOVERED`: original July 3 runtime DB/export artifacts were recovered and verified.
- `REGENERATED_ACCEPTED`: original artifacts were unavailable, but Phase 19 was regenerated from real FMP data in a healthy runtime.
- `PARTIAL_BLOCKED`: rebuilds ran, but strict research-cycle dry run still blocks with an exact blocker.
- `BLOCKED_INFRA`: runtime cannot support recovery/regeneration because of unresolved infrastructure issues.
- `BLOCKED_NO_DATA`: runtime is healthy, but required real bars/quotes are unavailable and FMP refresh cannot run.

## Guardrails

- Do not destroy any database volume without explicit operator confirmation.
- Do not print secrets or full credential-bearing URLs.
- Do not call FMP unless a bounded refresh is required and `FMP_API_KEY` is supplied only through the runtime environment.
- Do not bypass stale-window gates or weaken freshness checks.
- Do not use `allow_stale=true` except as a clearly labeled diagnostic after strict dry run blocks.
- Do not activate or deploy models.

## Recovery First

Before rebuilding anything:

1. Search local repo paths, ignored files, likely backup locations, and Docker volumes for Phase 19 SQLite DBs, exports, logs, model artifacts, replay IDs, and research cycle IDs.
2. Search for the committed Phase 19 export filenames and expected replay/research IDs.
3. If artifacts are recovered, verify file hashes/source IDs/artifact versions where possible.
4. If artifacts are absent, document exact missing evidence.

## Runtime Repair

Non-destructively inspect and repair:

1. Python: verify `python3.14` and backend venv versions. Use Python `3.14.6` only if available; otherwise document drift and use `3.14.4` as a non-acceptance fallback for tests.
2. Redis: identify the owner of host port `6379`; do not stop it without operator approval. If Redis is optional for the current evidence task, continue without claiming Redis verified.
3. Postgres: inspect schemas, tables, search path, Alembic state, and failed migration side effects. Perform only safe, non-destructive repair; stop if a volume reset would be required.

## Regeneration Conditions

Regenerate Phase 19 only if recovery fails and real data is available:

1. Confirm persisted real bars/quote snapshots exist.
2. If data is absent and `FMP_API_KEY` is present, run a bounded seed/incremental refresh with header-only auth.
3. If data is absent and no runtime key is present, mark `BLOCKED_NO_DATA` unless another real data source is recovered.
4. Rebuild in strict order: dirty-window audit, features, candidates, labels, replay, freshness check, strict research dry run, diagnostic stale comparison only if needed, exports, tests, scans.

## Reporting

Create Phase 19B reports for:

- evidence recovery;
- runtime repair;
- regeneration results;
- final completion and certification status.

Update Phase 19 docs and `docs/HANDOFF.md` with the final Phase 19B audit status.
