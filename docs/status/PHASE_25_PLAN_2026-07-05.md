# Phase 25 Plan

`PHASE_25_PLAN_STATUS = SCORER_DIAGNOSTICS_ONLY`

Phase 25 is scorer coverage diagnostics and out-of-sample selection-failure analysis for the Phase 24 15min Ten-AM specialist slice. It is not a model activation, model proposal approval, gate-loosening, threshold-selection, or broker-execution phase.

## Objectives

- Explain why the inactive Phase 24 scorer selected only 2 OOS `TAKE` candidates and 0 `WATCH` candidates from 145 OOS scored candidates.
- Attribute suppression to concrete persisted score-audit fields, evidence cells, scorer config gates, warnings, and penalties.
- Compare the entire OOS base cohort with scorer-selected and suppressed groups without promoting any OOS-derived filter.
- Diagnose whether the prior Phase 23 9-candidate result was likely a low-sample artifact after Phase 24 expansion.
- Recommend the next pre-registerable research step while keeping active models at `0`.

## Required Boundaries

- Do not activate models or approve proposals.
- Do not loosen validation, calibration, sensitivity, stale-data, or freshness gates.
- Do not select new thresholds from OOS outcomes.
- Do not use realized same-bar ambiguity, labels, or replay outcomes as live filters.
- Do not write fixtures to the evidence DB.
- Do not use broker execution, order routing, options data, production WebSocket ingestion, black-box ML, or self-learning claims.
- Do not expose `FMP_API_KEY` or place API keys in query strings.

## Work Sequence

1. Preflight the clean evidence DB, isolated test DB, fixture guard, active-model count, dirty-window count, Phase 24 docs/artifacts, and tracked env-file status.
2. Audit the score distribution for all Phase 24 OOS scored candidates.
3. Aggregate suppression reasons and top suppression pathways.
4. Analyze evidence-cell hierarchy coverage and sparsity for the 15min `ten_am_reversal_zone` cohort.
5. Compare OOS base, selected `TAKE`, selected `WATCH`, suppressed, top-score quartile, top-score decile, and Phase 23-like groups.
6. Run diagnostic-only threshold/config coverage checks using current config and training-only score quantiles. No production config is changed.
7. Compare Phase 23 and Phase 24 candidate definitions, selections, replay outcomes, and sensitivity behavior.
8. Generate hashed exports with source IDs and row counts.
9. Run required tests, scans, and final evidence audit.
