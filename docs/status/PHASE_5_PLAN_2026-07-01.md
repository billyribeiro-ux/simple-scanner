# Phase 5 Plan - PostgreSQL Repository Runtime Implementation And Parity Testing

Date: 2026-07-01

## Objective

Implement a PostgreSQL-backed repository runtime behind the existing repository registry contract and prove that the persisted API vertical slice works against both SQLite local fallback and PostgreSQL/TimescaleDB.

## Scope Guardrails

- No frontend UI work unless a response-shape type break requires it.
- No broker execution, order routing, options, gamma, Greeks, IV, market internals, or WebSocket work.
- No ML calibration or quant methodology expansion.
- No profitability, self-learning, or live-trading claims.
- No secret printing, logging, persistence, or documentation.

## Implementation Sequence

1. Preserve the existing repository contract
   - Keep API and service code backend-agnostic through `get_repository_registry()`.
   - Avoid route-level backend branching.
   - Preserve existing method names and return shapes for symbols, bars, features, candidates, labels, validation reports, models, active models, scanner runs, live signals, provider requests, exports, and daily reviews.

2. Add explicit backend selection
   - If `DATABASE_URL` is missing, use SQLite `data/local_repo.sqlite3` or `AMD_SQLITE_PATH`.
   - If `DATABASE_URL` starts with `sqlite:///`, use SQLite at that configured path.
   - If `DATABASE_URL` starts with PostgreSQL, use a Postgres runtime through synchronous SQLAlchemy/psycopg.
   - If Postgres initialization fails, fail loudly unless `AMD_ALLOW_SQLITE_FALLBACK=true`.
   - If explicit fallback is enabled, use SQLite and report `sqlite-fallback-from-postgres` with a non-secret reason.

3. Implement Postgres storage plumbing
   - Add a Postgres store that creates a synchronous SQLAlchemy engine using redacted/sanitized descriptors.
   - Convert asyncpg-style URLs to psycopg URLs for sync runtime.
   - Verify required tables/migration state at startup instead of silently creating production tables.
   - Provide safe `ping()` and `info()` status without leaking URLs or passwords.

4. Share repository behavior where practical
   - Reuse payload normalization, stable IDs, coercion helpers, and method-level business behavior.
   - Add store-level execution helpers for placeholder conversion, JSON serialization/deserialization, transactions, and row mapping.
   - Keep bulk writes transaction-scoped and avoid row-by-row commits for bars, features, labels, and signals.

5. Strengthen DB inspection
   - Keep `make db-migrate` authoritative.
   - Expand `make db-inspect` to verify expected tables, selected columns, unique constraints, critical indexes, JSON columns, Timescale extension, and Alembic revision.
   - Fail clearly on missing schema artifacts.

6. Parameterize API smoke
   - Refactor the existing smoke helpers so the same route workflow can run with SQLite and Postgres settings.
   - Add `make api-smoke-sqlite`, `make api-smoke-postgres`, and keep `make api-smoke` as the documented SQLite default.
   - Ensure the Postgres smoke skips honestly when Docker/Postgres is unavailable and passes when healthy.

7. Add repository parity tests
   - Use one deterministic dataset and assertions against SQLite and Postgres.
   - Check counts, active model behavior, scanner signals, export metadata, daily reviews, and reinitialization survival.
   - Cover backend selection cases: missing `DATABASE_URL`, SQLite URL, Postgres URL, failed Postgres without fallback, and failed Postgres with fallback.

8. Update security checks
   - Verify `/health`, `/config`, and `make doctor` report safe backend fields only.
   - Verify provider metadata and exports do not contain FMP keys or database URLs.
   - Run the existing secret scan without echoing the supplied FMP key.

9. Documentation and handoff
   - Update runtime, persistence, data model, API smoke, performance, README, and handoff docs.
   - Create Phase 5 completion and Postgres parity status docs.
   - State exactly what is safe to trust and what remains partial.

10. Verification
   - Run the full practical command matrix: backend gates, Docker/migration/inspection, SQLite/Postgres smoke, repository parity, frontend gates, compileall, diff check, and secret scan.

## Completion Criteria

Phase 5 is complete when SQLite and Postgres repository runtimes both pass the persisted API smoke workflow, backend selection reports honestly, Postgres never silently falls back to SQLite without `AMD_ALLOW_SQLITE_FALLBACK=true`, schema inspection verifies migration artifacts, security scans pass, docs reflect the true runtime state, and all practical quality gates pass.
